# AI Project Maintenance Standard

**Version:** 1.0

**Status:** Foundation Document

**Applies To:** All Reticle Systems projects

---

# Purpose

Software should not only be maintainable by humans.

It should be maintainable by AI agents.

As projects grow, AI productivity naturally decreases unless the repository is actively maintained.

The objective of this document is to define the standards required to keep a project understandable, navigable and safe for both human developers and AI agents.

A healthy repository produces:

* faster implementation
* fewer hallucinations
* better architectural consistency
* safer autonomous changes
* lower maintenance costs
* easier onboarding for new developers and AI agents

---

# Philosophy

The project itself is part of the AI system.

Poor documentation, stale files and unclear ownership reduce AI quality regardless of which language model is used.

Improving repository quality often produces greater gains than changing models.

---

# AI Maintenance Principles

Every repository should optimise for:

* clarity
* discoverability
* ownership
* repeatability
* verification
* documentation
* recoverability

---

# AI Readiness

Every project should maintain a high AI Readiness Score.

The repository should be easy for a completely new AI session to understand within minutes.

AI should never need to guess:

* where the truth lives
* which files are generated
* which files should be edited
* how to build
* how to test
* how to deploy

---

# Repository Health Areas

ODIN should periodically assess:

## Documentation

Questions:

* Is the Source of Truth current?
* Does documentation reflect reality?
* Have tutorials been updated?
* Are obsolete documents archived?

---

## Architecture

Questions:

* Does the code still match the documented architecture?
* Have major design decisions been recorded?
* Are responsibilities clearly separated?

---

## Ownership

Every significant area should have an identified source owner.

Example:

Authentication

Source:

```
src/auth/
```

Generated:

```
Never edit generated authentication artifacts directly.
```

The repository must clearly distinguish between:

* editable source
* generated output
* build artefacts
* legacy compatibility files

---

## Generated Files

Generated files should always be identified.

AI should edit source files whenever possible.

Examples include:

* generated API clients
* compiled assets
* build outputs
* ORM-generated code
* autogenerated documentation

The source responsible for generation must be documented.

---

## Legacy Code

Old compatibility layers should be marked.

Each legacy component should state:

* why it exists
* whether it can be removed
* what depends on it

Dead code should be removed whenever safe.

---

## Repository Cleanliness

Repositories should avoid:

* duplicate implementations
* abandoned experiments
* stale branches
* obsolete reports
* unused assets
* temporary files committed to source control

---

# AI Navigation

Every project should contain an AI navigation document.

Recommended filename:

```
AI_INDEX.md
```

or

```
ODIN_INDEX.md
```

The index should include:

* project overview
* architecture summary
* directory map
* source of truth location
* generated folders
* build commands
* test commands
* deployment process
* documentation map

Every AI session should read this first.

---

# Preflight Checklist

Before modifying code, every AI agent should perform a preflight check.

Minimum checklist:

✓ Read Source of Truth

✓ Read AI Index

✓ Identify ownership of affected files

✓ Confirm files are not generated

✓ Confirm current branch

✓ Understand build process

✓ Understand test process

✓ Review related documentation

No implementation should begin before preflight completes.

---

# Implementation Rules

During implementation:

* edit source files only
* avoid generated output
* preserve architectural consistency
* update documentation where necessary
* keep changes focused
* avoid unrelated refactoring unless approved

Large changes should be broken into incremental slices.

Every slice should remain buildable.

---

# Verification Standards

Every completed slice should verify:

* project builds
* tests pass
* lint passes
* formatting passes
* documentation updated where required

Verification must be automated whenever possible.

---

# Finish Gate

A task is not complete until the Finish Gate passes.

Required checks:

✓ Repository builds successfully

✓ Relevant tests pass

✓ Documentation updated

✓ Source of Truth updated if architecture changed

✓ AI Index updated if navigation changed

✓ No accidental generated file edits

✓ Git working tree reviewed

If the working tree remains intentionally dirty, the reason must be documented.

---

# Continuous Improvement

Maintenance is an ongoing activity.

ODIN should regularly propose improvements without waiting for new feature requests.

Examples include:

* reducing complexity
* improving naming
* simplifying architecture
* removing duplication
* improving accessibility
* increasing test coverage
* updating documentation
* removing obsolete dependencies

---

# Recovery Mode

If a repository has become difficult to maintain, ODIN should enter Recovery Mode.

Recovery Mode performs:

* documentation audit
* architecture audit
* ownership audit
* dependency audit
* generated file audit
* dead code detection
* repository cleanup recommendations

Recovery should prioritise the safest improvements first.

---

# AI Readiness Assessment

ODIN should produce an AI Readiness Report covering:

* Documentation
* Architecture
* Repository Structure
* Ownership
* Testing
* Build Process
* Prompt Readiness
* Generated File Separation
* Git Hygiene
* Maintainability

The report should include recommendations ranked by impact and implementation effort.

---

# Documentation Maintenance

Documentation should evolve with the project.

Whenever architecture changes:

* update Source of Truth
* update tutorials
* update AI Index
* update architecture documentation
* update implementation notes

Documentation should never significantly lag behind implementation.

---

# Long-Term Goal

The ideal Reticle Systems repository is one where any competent developer or AI agent can become productive within minutes.

Success is achieved when:

* no important knowledge exists only in someone's memory
* AI never has to guess where truth lives
* documentation reflects reality
* generated code is clearly separated from source
* implementation remains incremental and verifiable
* every completed task leaves the repository in a better state than it was found.

---

# Guiding Principle

> **Every AI interaction should improve the repository for the next AI interaction.**

The objective is not merely to complete today's task, but to leave behind a cleaner, better documented, more understandable project that accelerates every future contributor.
